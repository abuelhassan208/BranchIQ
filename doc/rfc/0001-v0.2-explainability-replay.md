## 2. Problem Statement

BranchIQ v0.1.0 provides robust, bounded deterministic runtime execution, immutable decision trees, mathematical scoring cores, beam-search pruning, and priority-first traversal. It generates debug snapshots containing the raw parameters, execution metadata, and traces.

However, developers and automated systems integrating BranchIQ still lack standard mechanisms to:
1. **Explain Decision Logic in Human-Readable Forms**: Developers must manually reconstruct why a decision was reached by parsing raw utility scores, impact metrics, and pruning logs.
2. **Compare Alternative Paths**: There is no programmatic path comparison tool to answer: *"Why was Path A chosen over Path B?"* or *"At which node did Path B get pruned or outscored?"*
3. **Inspect and Query Historical Decisions Offline**: Reconstructing a decision in post-mortem analysis currently requires complete tree execution, making offline analysis and debugging complex and error-prone.
4. **Perform Diffing of Snapshots**: When decision trees or configurations evolve, developers need a deterministic snapshot-diffing utility to compare two evaluations side-by-side.

### Practical Implications
* **Debugging**: Identifying why the engine made an unexpected decision requires inspecting multi-dimensional scores.
* **Trust & Compliance**: Regulated industries (e.g., finance, healthcare, security operations) require strict auditing of automated actions. Raw logs do not suffice as explainable audit trails.
* **Reproducibility**: Ensuring that production decisions can be verified on local systems or continuous integration pipelines requires replayable diagnostics.

## 3. Goals

The primary goal of BranchIQ v0.2 is to bridge the gap between deterministic runtime telemetry and developer usability.

1. **Deterministic Explanation Generation**: Provide a stateless, pure-logic utility to generate structured decision explanations. Given identical inputs, the explainer must yield identical explanations.
2. **Replay-Safe Diagnostics**: Design replay APIs that reconstruct decision nodes, traversed pathways, and pruner telemetry entirely from a `DebugSnapshot` without re-evaluating the tree.
3. **Structured Decision Reports**: Define an immutable, serializable `ExplanationReport` model that encapsulates selected path properties, discarded alternatives, scoring contributions, and traversal reasoning.
4. **Snapshot Comparison Tooling**: Deliver a diff system (`SnapshotDiff`) that compares two snapshots to reveal differences in scoring, pruning, and chosen paths.
5. **Stable Formatted Outputs**: Support deterministic Markdown and JSON exports of explanation reports and diffs to facilitate integration with CI/CD artifacts, logging databases, and reporting systems.

## 4. Non-Goals

To maintain the architectural integrity of BranchIQ as a high-performance, predictable runtime engine, the following are strictly excluded from v0.2:

1. **AI-Generated Explanations and LLM Integrations**: We will not integrate Large Language Models (LLMs) or generative text networks. The runtime reasoning must be mathematical, analytical, and literal.
2. **Natural Language Generation (NLG) Systems**: No template-free natural-language hallucination engines or adaptive text synthesis. Exposing the reasoning behind decisions will rely on structured tabular, mathematical, and deterministic rule-based formats.
3. **Async Replay or Distributed Execution**: The replay and explanation layer must run entirely synchronously on a single thread. No Future or Stream constructs will be introduced to runtime evaluation or post-evaluation processing.
4. **Visual Graph UIs and Flutter Widgets**: The package remains a pure-Dart library. Visual representations, interactive node graphs, or DevTools UI widgets are out-of-scope for the core package and will be deferred to downstream libraries.
5. **Runtime Mutation during Replay**: Replaying a session must never allow modifying tree nodes, score weightings, or pruning parameters. Replay is strictly read-only and diagnostic.
6. **Adaptive Reasoning and Auto-Optimization**: The engine will not dynamically alter its traversal strategy or heuristic weights during execution or replay based on the explanation reports.

## 5. Core Explainability Philosophy

The foundation of the BranchIQ Explainability layer is guided by a single principle:

> **"Explain only what the runtime actually evaluated."**

We enforce the following design rules:
1. **Zero Inferred Reasoning**: We never assume what an evaluation step meant. If a value was not recorded during the runtime cycle in the `DebugSnapshot` or `EvaluationResult`, it does not exist in the explanation.
2. **Zero Fabricated Logic**: Explanations must strictly reflect the mathematical contributions (probability, impact, cost, confidence) and the structural traversal steps of the engine.
3. **No Probabilistic Storytelling**: We do not construct narratives around decision outcomes. Instead, we present the raw mathematical justification (e.g., why a specific path scored higher than its competitors, which nodes were pruned and why).
4. **Traceability**: Every output of the explainer must map back to a specific component (e.g., the `BeamPruner`, the `Normalizer`, the `TraversalEngine`) via traces and numeric summaries.

## 6. Explanation Report Architecture

The `ExplanationReport` is the central output of the explainability system. It is an immutable, pure Dart model generated by analyzing an `EvaluationResult` or reconstructed from a `DebugSnapshot`.

### Architectural Structure
An `ExplanationReport` contains:
* **Selected Path Details**: The optimal path sequence, summarizing depth, name/ID, and step-by-step utility accumulation.
* **Selected Node Summaries**: Detailed metrics for each node in the optimal path.
* **Rejected Alternatives**: A list of nodes that branched from the optimal path but were outscored or bypassed, along with their final scores.
* **Pruning Summaries**: A deterministic map detailing which nodes were pruned, grouped by their pruner source (e.g., Beam, Probability, Score) and reasons.
* **Traversal Reasoning**: An analysis of the traversal priority queues, including details on explored vs. unexplored frontiers.
* **Score Contribution Analysis**: A mathematical breakdown showing the absolute and percentage contributions of `probability`, `impact`, `cost`, and `confidence` to the total path utility.

### Serialization & Exports
* **Stable JSON Export**: Stable, sorted serialization schemas representing the report.
* **Deterministic Markdown Export**: A clean, structured Markdown reporter designed for storage as Git artifacts or terminal output.

### Technical Blueprint (Dart Model Signature)

## 7. Node Explanation Model

The `NodeExplanation` model details the mathematical evaluation of a single node in the context of its parent and siblings.

### Core Metrics
* **Identity**: `nodeId`, `parentId`, `depth`.
* **Utility Parameters**: `probability`, `impact`, `cost`, `confidence`, and computed `score`.
* **Relative Contributions**:
  * **Probability Contribution**: The effect of the node's probability weight.
  * **Impact Contribution**: The normalized contribution of the impact value.
  * **Cost Contribution**: The cost deduction value.
  * **Confidence Contribution**: The effect of depth-decayed confidence.
* **State Categorization**:
  * **Pruning Status**: Excluded (with reason) or evaluated.
  * **Traversal Rank**: The order of exploration relative to other nodes on the same frontier.

### Stable Ordering
To guarantee deterministic serialization and diffing, `NodeExplanation` lists must always be sorted lexicographically by their `nodeId` when represented in maps, and topologically (by depth, then sibling order) when represented in paths.

### Technical Blueprint

## 8. Path Comparison System

When debugging, the most common question is: *"Why did the engine choose Path A (e.g., automatic_approval) instead of Path B (e.g., manual_review)?"*
The `PathComparisonSystem` compares the selected path with a user-specified rejected alternative, returning a `DecisionComparison` object.

### Comparison Logic
The comparison system performs the following analysis:
1. **Common Ancestor Analysis**: Identifies the branching node where the two paths diverged.
2. **Divergence Score Breakdown**: Compares the metrics of the direct children of the branching node.
3. **Step-by-Step Cumulative Tradeoffs**: Computes cumulative utility, cost, confidence, and probability differences.
4. **Pruning Audits**: Checks if the rejected path was excluded because a segment of it was pruned during beam-search or probability-pruning.

### Technical Blueprint
```dart
class DecisionComparison {
  final List<String> selectedPath;
  final List<String> targetPath;
  
  final String divergenceNodeId;
  final double selectedPathUtility;
  final double targetPathUtility;
  final double utilityDifference;

  // Branching node details
  final NodeExplanation selectedBranchWinner;
  final NodeExplanation targetBranchLoser;

  // Pruning audit
  final bool targetPathPruned;
  final String? targetPathPruneReason;

    required this.selectedPath,
```

## 9. Replay System Architecture

The Replay system's goal is to recreate the state of an evaluation run strictly from its saved `DebugSnapshot`.

```
                    ┌─────────────────┐
                    │  DebugSnapshot  │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │  ReplayLoader   │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │  ReplaySession  │
                    └────────┬────────┘
                             │
            ┌────────────────┴────────────────┐
            ▼                                 ▼
 ┌────────────────────┐            ┌────────────────────┐
 │  ReplayInspector   │            │ BranchIQExplainer  │
 └────────────────────┘            └────────────────────┘
```

### Strict Replay Restrictions
* **No Engine Execution**: The replay loader does not execute `BranchIQEngine.evaluateSync()`. It does not calculate scores, perform beam-pruning, or run traversal queues.
* **Data Reconstitution**: It reconstructs a static replica of the tree and decisions from `DebugSnapshot.nodeSnapshots` and `DebugSnapshot.selectedPath`.
* **Zero Overhead**: Because no scoring functions or rules are re-evaluated, replaying a session has minimal computational overhead.
* **Snapshot Completeness**: Replay operates as a closed system. It is independent of external state, runtime environments, or clock variables.

### Replay Components
1. **`ReplayLoader`**: Parses a JSON-serialized `DebugSnapshot` or direct object instance and initializes a `ReplaySession`.
2. **`ReplaySession`**: Holds the reconstructed immutable state representing the evaluation event.
3. **`ReplayInspector`**: Provides deterministic APIs to query the reconstructed state (e.g., fetching a node's metrics, verifying pruning decisions, reading traces).

### Technical Blueprint
```dart
class ReplaySession {
  final DebugSnapshot snapshot;
  final DecisionTree reconstructedTree;

  ReplaySession._(this.snapshot, this.reconstructedTree);

  List<String> get selectedPath => snapshot.selectedPath;
  List<String> get prunedNodeIds => snapshot.prunedNodeIds;
  List<String> get runtimeTraces => snapshot.runtimeTraces;
  List<String> get pruningTraces => snapshot.pruningTraces;

  /// Inspects a specific node's reconstructed attributes.
  DecisionNode inspectNode(String nodeId);
}

class ReplayLoader {
  /// Loads a [DebugSnapshot] directly into a read-only [ReplaySession].
  static ReplaySession load(DebugSnapshot snapshot) {
    // Reconstitute decision nodes from nodeSnapshots
    final nodes = <DecisionNode>[];
    snapshot.nodeSnapshots.forEach((id, data) {
      nodes.add(DecisionNode.fromJson(data));
    });
    
    final tree = DecisionTree.fromNodes(nodes);
    return ReplaySession._(snapshot, tree);
  }
}
```

* **Selected Path Changes**: Tracks if the optimal path changed between Snapshot A and Snapshot B.
* **Utility Score Shift**: Quantifies the variation in total accumulated utility.
* **Pruning Discrepancies**: Pinpoints nodes that were pruned in Snapshot A but evaluated in Snapshot B, or vice-versa.
* **Node Parametric Variance**: Compares scores and individual attributes (cost, impact, probability, confidence) for individual nodes evaluated in both snapshots.
* **Execution Telemetry Diff**: Compares metadata, runtime durations, and steps.

## 12. Markdown Export System

The Markdown Export system renders explanation reports and snapshot diffs into plain-text markdown documents. This enables deep integration with continuous integration (CI) environments, command line tools, and file-based audit trails.

### Layout Specification
Every generated markdown file must adhere to a strict structural schema. Sections are arranged in a predictable, stable order to ensure consistent layout across runs:

1. **Title and Execution Summary**: Version, runtime status, fallback status, execution duration.
2. **Optimal Path Walkthrough**: A sequential path trace (`root -> node_a -> node_b`) displaying accumulated utility.
3. **Core Utility Analysis**: A detailed table breakdown of chosen nodes containing impact, cost, probability, confidence, and total node score.
4. **Pruning Outcomes**: A table or list containing pruned node IDs, the pruner that performed the operation, and the mathematical or structural reason for the exclusion.
5. **Trace Logs**: Stable, chronological sequence of runtime execution logs.

### Deterministic Format Constraints
* **Numeric Precision**: All floating-point scores are formatted with exactly 4 decimal places (e.g., `0.8472`).
* **Table Sorting**: All node listings that are not sequential paths must be sorted lexicographically by their unique `id`.
* **Zero System Clock Dependency**: Generating the markdown report must not reference local times, execution environments, or system locales.

## 14. Replay Safety Guarantees

The BranchIQ Replay layer guarantees exact decision reconstruction. To achieve this, it implements several security and runtime constraints:

* **No Side-Effects**: Loading or querying a replay session must never modify global state or trigger network/disk access.
* **Strict Snapshot Read-Only Integrity**: Replay loaders operate on a copy of snapshot data. The source snapshot is treated as immutable, preventing post-decision tampering.
* **No Engine Evaluation Paths**: The replay session does not interact with the scoring normalizer, Beam pruning limits, or traversal queue builders. This prevents discrepancies caused by floating-point precision changes or compiler optimization shifts between evaluation and replay times.
* **Exact Diagnostic Equivalence**: Replaying a snapshot yields the exact same logical paths and scores as the live engine run.

## 15. Runtime Trace Integration

Traces generated during tree traversal, pruning, and scoring are captured sequentially inside `DebugSnapshot.runtimeTraces` and `DebugSnapshot.pruningTraces`. The Explainability layer integrates these traces into reports.

### Trace Structuring
* **Scoring Traces**: Detail normalizations, raw utilities, and applied weights.
* **Pruning Traces**: Record pruned nodes alongside their thresholds and actual score drops.
* **Traversal Traces**: Capture path selection steps and child evaluations.

To ensure trace lines are deterministic, they must be formatted using a strict template that excludes runtime memory addresses or random hashes. Traces are sorted chronologically by execution step sequence.


## 16. Failure Diagnostics

Replaying decisions depends on snapshot integrity. The system must detect and safely report corrupted or incomplete execution snapshots.

### Handled Failure States
1. **Invalid Schema**: The loaded JSON does not match the required schema version or misses critical fields (e.g., `selectedPath`, `nodeSnapshots`).
2. **Missing Node References**: The `selectedPath` references a node ID that has no matching entry inside the `nodeSnapshots` collection.
3. **Corrupted Replay State**: The snapshot has mismatched total utilities compared to the sum of the node metrics in the path.

### Error Handling Rules
* **No Silent Bypasses**: The loader will throw a highly descriptive, typed `ReplayCorruptException` containing the exact validation discrepancy.
* **Deterministic Messages**: Error messages must follow a strict template with no platform-specific details.

### Technical Blueprint
```dart
class ReplayCorruptException implements Exception {
  final String message;
  final String? missingNodeId;
  final String? mismatchReason;

  const ReplayCorruptException(
    this.message, {
    this.missingNodeId,
    this.mismatchReason,
  });

  @override
  String toString() => 'ReplayCorruptException: $message';
}
```

## 17. Performance Constraints

The explainability and replay structures are designed to be used in production diagnostics and CI/CD pipelines. They must operate within strict resource boundaries:

1. **Bounded Replay Memory**: Replay memory footprint is bounded to `O(N)` where `N` is the number of evaluated nodes in the snapshot. The loader must never retain or cache external data.
2. **Bounded Diff Complexity**: Snapshot diffing operates in `O(A + B)` time where `A` and `B` represent the node counts in the source and target snapshots, respectively.
3. **Bounded Markdown Generation**: Generating a markdown document must execute in a single linear pass over the report nodes.
4. **No Thread/Isolate Dependencies**: Operations must not spin up background Isolates or native platform threads, preserving execution efficiency on single-threaded runtimes.

## 18. Testing Strategy

Robust testing of explainability and replay components is critical to ensure their output remains deterministic.

### Core Testing Pillars
1. **Deterministic Assertions**: Verify that running `BranchIQExplainer.explain(result)` on identical evaluation inputs produces byte-for-byte identical JSON and Markdown outputs.
2. **Replay Accuracy Check**: Test that replayed sessions reconstruct the exact same path and utility values as the original live engine execution.
3. **Corrupt Snapshot Isolation**: Assert that `ReplayLoader.load()` throws a `ReplayCorruptException` when loaded with snapshots that are incomplete or have mismatched schemas.
4. **Stable Diff Tests**: Verify that `SnapshotDiffer.compare()` produces deterministic diff objects regardless of the insertion order of nodes in the original snapshot map.
5. **Golden File Comparisons**: Maintain golden markdown templates in the codebase. Verify that generated markdown matches the golden templates exactly.

## 19. Migration & Compatibility

As BranchIQ transitions from v0.1.0 to v0.2.0, backward compatibility with older debug snapshots must be preserved.

### Schema Versioning Policies
* **Backward Compatibility**: `ReplayLoader` must accept older snapshots produced by v0.1.0. If older snapshots lack modern properties (e.g., specific traversal metrics or confidence decay breakdowns), the loader will substitute default values (`0.0` or empty placeholders) instead of throwing an error.
* **Forward Compatibility**: Modern v0.2.x snapshots will contain structural schema versions (`"schemaVersion": "2.0"`). Older parsers attempting to load newer snapshots must reject them if structural differences exist.
* **Stable Upgrades**: When new properties are added to `DecisionNode` in future versions, the snapshot serialization schema must include optional properties, maintaining backward compatibility.

## 22. Risks & Complexity Analysis

| Identified Risk | Severity | Mitigation Strategy |
| :--- | :--- | :--- |
| **Explanation Drift**: The explainer calculates contributions differently than the runtime engine due to code updates. | High | Centralize normalizations in `src/math` and verify mathematical alignment using daily automated integration tests. |
| **Replay Inconsistency**: A replayed session yields different path outcomes than the original run. | High | Restrict replay from executing traversal or scoring. Replay is strictly read-only, reading directly from the snapshot. |
| **Serialization Instability**: Map key ordering varies across Dart compilers, breaking diffs. | Medium | Enforce alphabetical sorting for all keys in JSON maps during serialization. |
| **Snapshot Size Bloat**: High execution depths yield extremely large snapshots. | Medium | Apply pruning policies at the engine level to limit snapshot sizes. |

## 23. Deferred Systems

The following systems are deferred to future versions to maintain focus on core library stability:

1. **Visual Tree Inspector and Flutter DevTools Extension**: Deferred to a standalone `flutter_branchiq_devtools` package. This keeps the core package lightweight and dependency-free.
2. **Real-time Live Runtime Streaming**: Streaming execution steps out of process during evaluation is deferred until there is a clear demand for external telemetry collectors.
3. **Distributed Replay and Network Collectors**: Automated uploads of snapshots to central logging servers are deferred. Integrators can implement this externally using the JSON output format.
4. **AI-Assisted Natural Language Insights**: Generating human-like text commentary on evaluations is deferred to downstream libraries.

### Architectural & Execution Details
1. **Most Critical Architectural Decision**: The choice to make the `ReplaySystem` snapshot-driven without re-executing any code. This guarantees perfect reproducibility across platforms and builds, shielding diagnostics from external state or calculation shifts.
2. **Biggest Replay Risk**: Ensuring that custom application state inside the `metadata` map does not introduce breaking changes or type mismatches when loaded onto different platforms.
3. **Most Difficult Implementation Area**: The `SnapshotDiff` engine. Calculating, aligning, and sorting complex differences in tree structures and metrics requires precision.
4. **Recommended Implementation Order**: 
   * Begin with Phase A (`NodeExplanation` and `ExplanationReport`) to establish stable models.
   * Proceed to Phase B (`ReplayLoader` and `ReplaySession`) to support offline diagnostic reconstruction.
   * Build Phase C (`SnapshotDiffer` and `DecisionComparison`) on top of the Replay session.
   * Finish with Phase D (`MarkdownExporter` and `JSONExporter`) once the underlying structures are finalized.
5. **Systems Intentionally Deferred**: Visual widgets, Flutter DevTools integration, live execution streams, and AI-assisted commentary are deferred to preserve the engine's lightweight, dependency-free core.
